# 下一步建议（Roadmap）

> 目标：在保持“便携/低资源占用”的前提下，尽快把产品从“可用”推进到“可长期使用”。

## 优先级 P0（建议立即做）

### 1) 金额与流水（里程碑 3）
- 收款流水：新增/编辑/删除
  - 类型：定金 / 尾款 / 其他
  - 金额输入：元（可小数），存储：分（整数）
  - 允许负数（退款）
- 金额调整：新增/删除（delta + reason）
- 详情展示计算值：总金额（当前）/ 定金汇总 / 已收 / 尾款或多收款

验收：
- 一笔定金 + 多笔收款 + 多次改价后，“尾款/多收款”计算正确且不丢历史。

### 2) 备份/恢复（里程碑 5）
- 导出：生成 zip（包含 `data/orderly.db` + `data/orders/`）
- 导入：先备份当前 `data/`（重命名为 `data.bak.<ts>/`）再替换

验收：
- 任意时刻导出后可在另一台机器/另一个目录恢复，数据一致。

## 优先级 P1（建议本周内做）

### 3) 订单质量提升
- 列表分页/虚拟滚动（订单量大时更顺滑）
- 详情页增加“复制微信号/复制仓库 URL”（可选）
- 统一错误提示（Dialog 表单校验/后端错误显示）

### 4) 测试增强（当前为“纯前端”）
- 扩展 Playwright 用例：
  - 编辑单子（保存后更新时间变化、列表顺序变化）
  - 日期筛选（范围命中/不命中）
  - Dialog 内容过高时仍可滚动（不贴边）

## 优先级 P2（后续可选）

### 5) 桌面端 E2E（更贴近真实）
- 需要时引入 `tauri-driver + WebDriverIO` 做真实桌面端到端（验证打开文件/打开目录等系统行为）。
