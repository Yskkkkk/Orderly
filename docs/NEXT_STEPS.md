# Orderly 下一步建议（MVP 完成后）

> 更新时间：2026-02-19
> 当前状态：**MVP 全部里程碑（1-6）均已完成**，包括核心订单 CRUD、金额流水、备份/恢复、附件树、前端 E2E 测试。
> 目标：在"便携/低资源占用"前提下，把产品从"可用"推进到"好用、可长期稳定使用"。

---

## P0 — 立即建议（稳定性与体验兜底）

### 1. 统一错误提示与表单校验

**现状**：후端命令执行失败时，前端尚无统一的错误展示机制。

**建议**：
- 在 `src/lib/` 中封装 `invoke` wrapper，统一捕获 Tauri 命令异常并在 UI 上以 Toast/Alert 呈现。
- Dialog 表单（新建/编辑订单）增加前端校验提示（如：仓库 URL 格式不合法时，内联提示而非静默保存）。

**验收**：后端返回错误时，用户能看到明确的中文提示，而不是静默失败或控制台报错。

---

### 2. 扩展 Playwright 测试用例

**现状**：已有基础用例（列表加载/新建单子/状态筛选），但编辑、日期筛选、金额流水等场景未覆盖。

**建议**（按优先级）：
1. **编辑单子**：保存后列表更新时间变化，验证 `order_update` 联动。
2. **日期筛选**：范围命中与不命中两个 case。
3. **金额流水**：新增一笔定金 → 验证"总金额/已收/尾款"计算正确。
4. **Dialog 内容溢出**：表单字段较多时，弹窗内可滚动且不贴边。

**验收**：`pnpm test:e2e` 全部通过，覆盖率覆盖到主要业务流。

---

## P1 — 本周内建议（体验打磨）

### 3. 列表性能优化（虚拟滚动 / 分页）

**现状**：全量渲染订单列表；订单量大时（≥ 500 条）可能出现卡顿。

**建议**：
- 引入 `@tanstack/react-virtual` 做虚拟滚动，或在后端 `orders_list` 支持 `limit/offset` 分页。
- 推荐先做**后端分页**（改动最小），配合前端"加载更多"或页码控件。

**验收**：2000 条订单下，列表滚动流畅，无明显掉帧。

---

### 4. 列表 / 详情体验细节

**建议**：
- **复制快捷按钮**：微信号、仓库 URL、订单 ID 旁增加一键复制按钮（`navigator.clipboard`）。
- **状态直接切换**：详情页状态字段支持下拉直接切换（无需打开编辑弹窗），并立即更新 `updated_at`。
- **空状态页**：列表无结果时显示友好的空状态插图+文案，而非空白区域。

---

### 5. 订单软删除（可选保留历史）

**现状**：技术规范 v1 无删除功能，但实际使用中会有需要"作废"某条订单的场景。

**建议**：
- 在 `orders` 表增加 `deleted_at_ms INTEGER` 字段（为 NULL 表示未删除）。
- 列表默认过滤已删除，增加"垃圾桶/已删除"视图。
- 彻底清除需二次确认。

**验收**：误操作后可在"已删除"中找回订单，彻底删除需二次确认弹窗。

## P2 — 后续可选（增强检索与发布）

### 6. 全文检索升级（FTS5）

**现状**：搜索只覆盖微信号/用户名/系统名称（普通索引）。

**建议**（对应 TECH_SPEC 方案 B 预留扩展点）：
- 在 SQLite 中创建 FTS5 虚拟表，索引 `system_name`、`tech_stack`、`deliverables`、`note`。
- 搜索框结果同时返回 FTS5 匹配结果，高亮匹配片段。

---

### 7. 桌面端 E2E 测试（tauri-driver）

**现状**：Playwright 用例跑的是 Web 模式（Mock Backend），无法验证"打开文件/打开目录"等系统行为。

**建议**：
- 参考 [tauri-driver 文档](https://tauri.app/v1/guides/testing/webdriver/) 配置 WebDriverIO。
- 优先覆盖：附件点击打开、打开单子目录、备份导出文件落盘。

---

### 8. 一键打包分发（CI/CD）

**现状**：需手动运行 `pnpm tauri build`，产物目录需手动整理。

**建议**：
- 使用 GitHub Actions + `tauri-action` 自动构建 Windows `.msi` / portable zip。
- 产物中自动包含 `data/` 占位目录与示例目录结构说明 `README.txt`。
